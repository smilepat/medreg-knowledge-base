## 1. 목표

### 프로젝트 목표 (큰 그림)
의료기기 정부규정 수백 페이지를 **LLM 친화 구조화 지식베이스**로 가공하여,
RAG(검색증강생성) 기반으로 **보고서·입찰제안서의 규정 준수**를 자동 지원한다.
핵심 철학: "PDF를 LLM에 통째로 던지기"가 아니라, 조항 하나하나를 색인 카드로
잘라 라벨(메타데이터)을 붙여 정리한다. **가공(Phase 1)에 가치의 90%**가 있다.

### 이번 세션 목표
1. GitHub 레포를 만들어 문서·개발의 기반으로 삼는다.
2. 첫 대상 규정을 정한다.
3. **문서 1건으로 파이프라인 전체(추출→구조화→검색→인용)를 끝까지 한 번 통과**시킨다(PoC).

## 4. 향후 과정

### 갈림길 A — 실제 인프라 부착 (로컬 → 운영)
- `embed.py`: Supabase pgvector 적재 + 임베딩(gemini-embedding-001 후보) →
  `search.py`에 의미검색 측 합산 = **하이브리드 검색 완성**.

- **필요:** Supabase DB 비번(ai-english-platform `rpxvlskmlrhivbiamicb` + 별 스키마 `medreg`),
  Gemini API 키.

- #2 허가·신고·심사 규정(고시 2026-6호): 표/별표 많음 → `extract.py` 표 경로 시험.

- 배치 파이프라인화(전 규정 일괄 처리), 버전관리(시행일) 체계.

### 갈림길 C — 입찰 컴플라이언스 매칭 (M3, 후순위)

### 결정 대기 사항
- A로 갈지 B로 갈지 (또는 PoC 완결 상태로 잠시 멈춤).
- 임베딩 모델: gemini-embedding-001(한국어 양호, 키 필요) vs 로컬 모델(키 불필요).
