> 의료기기법 본법으로 대조한 결과. 결론: **구조 충실도 높음. 단, 소스마다 기본 '버전'이 달라 버전관리가 필수.**

## 2. 구조 충실도 (= legalize-kr 파싱 품질)

- **제1조 본문 = 단어 단위 동일** (핵심 문구 일치 확인). → legalize-kr 파싱은 정확.
- **공통 조문 88개 중 제목 불일치 단 2개** — 그나마 둘 다 **실제 개정 내용**이지 파싱 오류 아님:
  - `제15조의2`: "희소ㆍ긴급도입…" → "긴급도입…" (제명 변경)
  - `제28조`: "품질관리심사기관의 지정 등" → "제조 및 품질관리체계에 대한 적합성인정" (개정)
- 조문 번호 차이도 **전부 실제 개정**(파싱 누락 아님):
  - 공식(2025-08-01)에만: `제41조, 제46조의2, 제53조` (미래본에서 삭제/이동)
  - legalize(2026-12-10)에만: `제2조의2, 제18조의6, 제28조의2~4` (신설 조문)

→ **결론: legalize-kr는 위계·본문을 충실히 보존한 고품질 파싱.** 두 소스의 차이는 *데이터 오류가 아니라 버전 차이*.

## 3. 핵심 교훈 — 버전관리는 선택이 아니라 필수

이번 대조가 우리 STRATEGY의 **버전관리(시행일 기준 현행 추적)** 원칙이 옳음을 실증했다:

- **소스마다 "기본 버전"이 다르다.** legalize-kr=최신 공포본(미래 포함), 공식 API=호출 시점 본.
  같은 "의료기기법"이라도 **어느 시행일 버전인지 명시하지 않으면 서로 다른 조문을 인용**하게 된다.
- **"현행"의 정의:** `시행일 ≤ 오늘`인 가장 최신 본. 미래 시행본(2026-12-10)은 *시행예정*으로 구분해야 하며,
  **현행으로 오인해 인용하면 사고**. (우리 환각방지·버전관리가 막아야 할 바로 그 케이스.)

## 4. medreg에 반영할 액션

1. **`is_current` 계산을 정교화** (현재 `chunk.py`는 `True` 하드코딩 — 이번 발견으로 부적절 판명):
   `is_current = (effective_date ≤ 오늘)`. 미래 시행본은 `is_current=false` + `status="시행예정"`.
2. **`fetch_law.py`에 시행일(efYd) 옵션 추가** — 어느 버전을 받을지 명시적으로 제어
   (기본=현행, 옵션=특정 시행일/최신 공포본).
3. **현행 + 시행예정 동시 보유 검토** — 입찰·보고서가 "시행 예정 규정 대비"를 요구할 수 있으므로,
   두 버전을 함께 적재하고 `status`로 구분(검색 기본은 현행).
4. **2차 소스 채택안:** legalize-kr는 **IP·키 없이 고품질**이라 보조 소스로 충분.
   단 *최신 공포본 편향*이 있으니, **권위·현행 정확도는 공식 API를 정본**으로, legalize-kr는
   폭넓은 커버리지·교차검증용으로 사용.
