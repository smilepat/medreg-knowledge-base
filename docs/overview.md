## ① 문제 정의 (지금 풀고 싶은 문제)

**현장의 문제:** 의료기기 기업(예: 대화기기)이 **보고서·입찰제안서**를 쓸 때, 적용되는
정부 규정이 `의료기기법 → 시행령 → 시행규칙 → 식약처 고시` + `조달·입찰 규정`으로
수백 페이지에 흩어져 있다. 담당자가 매번 이를 **수작업으로 찾아 인용·대조**해야 한다.

이때 생기는 구체적 고통:
- **시간:** 어떤 조항이 이 사안에 적용되는지 찾는 데 오래 걸린다.
- **정확도:** 조항을 잘못 인용하거나, **폐지·개정된 옛 조항**을 인용하는 사고.
- **LLM 직접 사용의 위험:** ChatGPT/Claude에 통째로 물으면 **없는 내용을 지어내거나
  (환각), 판례·조항을 잘못 인용**한다. (외부 실측 근거: `legalcheck` 레포 — LLM이 직역을
  놓치고 판례를 매번 다르게 오인용.)

**풀고 싶은 것(목표):** 규정을 한 번 잘 가공해 **재사용 가능한 지식 자산**으로 만들고,
질문하면 **출처(법령명+조항+시행일)를 인용해** 근거 있는 답만 주며, 근거가 없으면
**"확인 불가"**라고 솔직히 말하는 시스템. → 1차 타깃은 **보고서용 Q&A 검색**.

**핵심 제약(반드시 지킬 것):** 환각 방지(근거 있을 때만) · 버전 관리(현행 추적) ·
HITL(최종은 사람 검토). 틀리면 법적 책임이 따르는 영역이기 때문.

## ② 기존 데이터 검색 — 지금은 어떻게 찾나 (한계)

| 현재 방식 | 한계 |
|-----------|------|
| **law.go.kr 수동 검색** | 사람이 일일이 찾음. 본문은 JS로 가려 스크립트 수집도 막힘. 사안↔조항 연결은 머릿속에서 |
| **PDF/HWP를 LLM에 첨부** | 표 깨짐, 긴 문서 잘림, 출처 불명확, 환각 |
| **범용 LLM 질의** | 직역 누락·판례 오인용(검증됨), 시행일 무시 |

**기존 GitHub 레포 조사(우리가 직접 분석):** [reference-korean-medical-law-repos.md](reference-korean-medical-law-repos.md)
- 비슷한 시도: `medical_law_search_chatbot`(LangChain+Chroma 기본 RAG, **글자수 분할**·단일 docx·메타데이터 없음), `medical-compliance-tester`(멀티에이전트 RAG).
- 데이터셋: **`legalize-kr`/`admrule-kr`**(법령 Markdown, **IP·키 없이** raw fetch, 의료기기법 포함), `radsafety-laws`(주간 CI 자동 갱신 모델).
- 룰 기반: `legalcheck`(YAML 룰 정적분석, Apache-2.0).

**우리 차별점:** 글자수 분할이 아니라 **법령 위계(조/항/호)로 청킹** + **메타데이터(시행일·규정유형·출처)** + **버전관리** + **인용 강제/환각 방지**. → 같은 RAG라도 규정 준수용으로 신뢰 가능하게 만든다.

## ③ 법령 가져오기 (Ingestion)

**정본 경로 — 국가법령정보 Open API** (`scripts/fetch_law.py`):
- `type=XML`로 받으면 **조/항/호 위계가 이미 구조화**돼 옴(기본정보/조문/부칙/별표/개정문).
- 인증: OC 키 `smilepat` + 호출 PC IP 등록(`.env`의 `LAW_OC`). 권위 있고 시행일 정확.
- 검증: 의료기기법 전문(시행 2025-08-01, 조문 106) 수신 완료.

**보조 경로(검토 중) — legalize-kr raw fetch:**
- **IP·키 불필요**, 의료기기법 포함 폭넓음, git으로 개정 추적.
- 단, 가공본이라 **공식 API/law.go.kr 원본 대조 필요** → "공식=정본, legalize-kr=폭넓은 2차"로 이중화 검토.

**외부 문서(입찰 공고·회사 자료) — `scripts/extract.py`:**
- API에 없는 PDF/HWP는 pdfplumber로 **표를 Markdown으로 보존**하며 추출(스캔본은 OCR 신호 감지).

## ④ 처리해서 GitHub repo에 저장 + 가공

원문을 **이 레포 안에서** 단계적으로 가공해 재사용 자산으로 만든다(가치의 90%가 여기).

```
raw/<법령>.xml            ← ③에서 받은 원문 (gitignore: 용량·저작권)
   │ structure.py          위계 파싱: 장(章) 헤더 vs 조문 구분, 항/호/가지조
   ▼
processed/<법령>_structured.{json,md}   ← 기계용 JSON + 사람 검수용 MD (gitignore: 중간물)
   │ chunk.py              조 단위 contextual 청킹 + 메타 + 정의어/교차참조
   ▼
samples/<법령>_processed.json   ← ★ 가공 완료 지식 자산 (git 추적·공유)
   │ search.py             로컬 검색(키워드+글자유사도) + 메타필터
   │ report.py             출처 인용 답변 + 환각방지 + 버전/HITL 고지
   ▼
(향후) Supabase pgvector 적재 + 임베딩 → 의미검색 = 하이브리드 완성
```

**무엇을 "가공"하나(=색인 카드 만들기):**
- **청킹:** 조항 1개 = 청크 1개. 청크 안에 `[법령명 제N조(제목)] 장` 맥락을 넣어 자기완결.
- **메타데이터:** 법령명·규정유형·조항·시행일·현행여부·출처URL·교차참조·정의어.
- **사전화:** 정의 조항(제2조 등) → 용어 사전 분리 / "제20조에 따라" → 교차참조 링크.

**GitHub repo에 저장하는 이유:** 가공 결과(`samples/`)와 파이프라인(`scripts/`)을 한곳에 두어
**다른 PC에서 clone 후 바로 재현**(`docs/handoff-2026-06-15.md`의 명령) + 버전 관리(개정 추적) + 협업 기반.
민감물(원문 대용량·회사/입찰 기밀)은 `.gitignore`로 제외.
