## 0. 한 줄 정의

정부 규정 수백 페이지를 LLM이 잘 쓰는 **구조화 지식베이스**로 만들어,
보고서·입찰제안서의 규정 준수에 활용한다. **가공 단계에 가치의 90%**가 있다.

## 1. 용어 (ELI5)

| 용어 | 설명 |
|------|------|
| RAG (검색증강생성) | LLM이 답하기 전 관련 문서를 찾아와(retrieve) 근거로 답하게 하는 방식. 환각 감소 |
| Chunking (청킹) | 긴 문서를 검색·인용하기 좋은 작은 조각으로 쪼개기 |
| Embedding (임베딩) | 글의 "의미"를 숫자 벡터로 변환. 키워드가 안 맞아도 뜻으로 검색 |
| Vector DB (벡터 DB) | 임베딩을 저장하고 의미가 가까운 조각을 빠르게 찾는 DB (= pgvector) |
| Metadata (메타데이터) | 조각마다 붙는 라벨(법령명·조항·시행일·적용대상 등) |
| HITL (사람 검토) | 최종 결과는 반드시 사람이 검토 (규정 준수 안전장치) |

## 2. 4단계 파이프라인

### Phase 0 — 범위 정의 ("무엇을, 무엇을 위해")
1. **어떤 규정인가** — 의료기기 위계: `의료기기법(법) → 시행령 → 시행규칙 → 식약처 고시/공고`
   + 조달 영역: `국가계약법령 / 조달청 입찰 규정 / 나라장터 요건`. **이 위계를 그대로 시스템에 반영.**
2. **산출물이 무엇인가**
   - **보고서:** 적용 규정 찾아 인용하는 *검색·요약* → **첫 타깃 (확정)**
   - **입찰제안서:** 공고 요구조건 충족 여부 대조하는 *컴플라이언스 매칭* → 다음 단계

### Phase 1 — 원문 → 구조화 (★ 가장 중요)
1. **추출(Extraction)** — 한국 정부 문서 함정: ① HWP 多 ② 표 多 ③ 스캔본(이미지 PDF) 혼재.
   → HWP 별도 변환 / 스캔본 OCR / 표 보존 전용 파서(Upstage·LlamaParse·Docling 비교 평가).
2. **정제(Cleaning)** — 머리말·꼬리말·페이지번호·줄바꿈 쓰레기 제거. 표는 Markdown 표로.
3. **구조화(Structuring)** — 법령 위계(조/항/호/목) 보존. 출력 = Markdown(계층 헤더) + JSON 메타데이터 이중 구조.
4. **청킹(Chunking)** — 조항 1개 = 청크 1개. 단 자기완결(contextual chunking: 상위 조 제목·법령명 포함).
5. **메타데이터 태깅** — `법령명/조항번호/시행일/규정유형/적용대상/원문출처`.
   + 정의어는 사전 테이블 분리, 교차 참조는 링크 연결.

### Phase 2 — 저장 + 검색 (RAG 엔진)
- **저장:** 청크+메타데이터를 pgvector(Supabase 재사용)에 적재.
- **하이브리드 검색:** 의미 검색(임베딩) + 키워드 검색("제15조", "GMP") **둘 다**.
- **메타데이터 필터링:** "현행만", "이 품목등급만" 라벨로 먼저 거른 뒤 검색.

### Phase 3 — 활용
- **보고서:** 질문 → 조항 검색 → **출처(법령명+조항) 인용**해 초안.
- **입찰제안서:** 공고 요구사항 항목화(requirement matrix) → 규정/회사자료 대조 → 충족/미충족/확인필요 + 근거 인용.

---

## 3. 3대 안전장치

1. **환각 방지** — 검색된 원문이 있을 때만 규정 주장. 근거 없으면 "확인 불가". 인용 `법령명+조항+시행일` 의무.
2. **버전 관리** — 시행일 기준 현행 추적. 폐지 조항 인용 방지.
3. **HITL** — 최종본은 담당자가 인용 출처 직접 확인.
