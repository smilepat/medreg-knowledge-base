## 도메인 핵심 원칙 (이 프로젝트에서 반드시 지킬 것)

### 1. 법령 위계 보존 (★ 가장 중요)
- 글자 수로 자르지 말 것. **법령 위계(조 / 항 / 호 / 목)를 천연 청킹 경계로 사용**.
- 청크 1개 = 원칙적으로 **조항 1개**.

### 2. Contextual Chunking (자기완결 청크)
- 각 청크는 *자기 자신만으로 이해*되어야 함.
- "제2항"만 떼지 말고, 청크 안에 `○○법 제15조 제2항 / 상위 조 제목`을 함께 포함.

### 3. 메타데이터 필수 필드 (모든 청크)
| 필드 | 설명 | 예 |
|------|------|----|
| `law_name` | 법령명 | 의료기기법 |
| `article` | 조항번호 | 제15조 제2항 |
| `effective_date` | 시행일 | 2025-01-01 |
| `reg_type` | 규정유형 | 법 / 시행령 / 시행규칙 / 고시 / 공고 |
| `applies_to` | 적용대상(품목·등급) | 2등급 의료기기 |
| `source` | 원문 페이지/링크 | p.12 / URL |
> 상세 스키마: `docs/metadata-schema.md`

### 4. 정의어·상호참조 처리
- "제3조 정의" 같은 용어 사전은 **별도 사전 테이블**로 분리.
- "제20조에 따른다" 같은 교차 참조는 **링크로 연결**.

### 5. 3대 안전장치
- **환각 방지:** 규정 주장은 검색된 원문이 있을 때만. 근거 없으면 "확인 불가". 인용엔 `법령명+조항+시행일` 의무.
- **버전 관리:** 시행일 기준 "현행" 추적. 폐지 조항 인용 금지.
- **HITL:** 최종본은 사람이 인용 출처 확인. LLM은 초안 작성자.

## 작업 방식 (전역 규칙 + 이 프로젝트 강조)

- **한 파일/한 스크립트씩** 만들고 검증 후 다음으로. 처음부터 수백 페이지 넣지 말 것.
- **문서 1개로 파이프라인 전체(추출→구조화→검색→인용)를 끝까지 통과**시킨 뒤 규모 확대.
- 한국어로 설명 + 영문 기술용어. 코드 주석도 한국어.
- 기술용어 첫 등장 시 한국어 설명 병기.
- 새 외부 SaaS 도입 자제. 벡터 DB는 기존 Supabase pgvector 재사용.

## 출력 포맷 표준
